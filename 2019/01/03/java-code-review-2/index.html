恶意文件执行

例：应用提供文件上传功能，并可通过指定路径访问。假如上传1.jpg之后便可通过http://www.xxx.com/files/1.jpg进行访问，那么自然会有人试图上传 backdoor.jsp，一旦服务器能够执行http://www.xxx.com/files/backdoor.jsp这个路径，那么就会出现渗透和攻击。

防范措施：

  1. 检查待上传文件的文件类型。
  2. 对服务器进行设置，或者对目录加以处理，防止服务器执行存放目录下的文件。

越权访问

例1：自己的工作单是http://www.xxx.com/business/view.jsp?id=1024，将1024改成其他数字就看到了其他人的工作单。

防范措施：加入访问控制检测，只允许查询自己的工作单。例如校验工作单的办理者是否为本人，或者在SQL上面加入权限相关条件，使不具备访问权限的数据查不出来。

例2：张三有权限查看一些敏感数据，而李四没有。李四发现张三查看敏感数据的网址是http://www.xxx.com/business/viewsecret.jsp，之后将其记录下来，并使用自己账号访问这个地址，结果也看到了敏感数据。

防范措施：假设用户知道每一个功能的URL。对于每个URL，进入之前程序需要先校验用户是否具有相关权限。另外为了方便，也可以实现用户访问控制相关框架。

例3：你在某功能上只有部分操作权限，使用浏览器的“检查元素”功能找到那些按钮，将disabled属性删除，结果便可以进行受限操作。

防范措施：无论前台是否有校验，后台务必进行校验。

信息泄漏

例1：登录系统，用户名错误时系统提示“用户名错误”，密码错误时系统提示“密码错误”。这样攻击者看到“密码错误”时便知道用户名是正确的。

防范措施：

  1. 提示“用户名或密码错误”。
  2. 加入验证码，或者多次失败后锁定账号，以防暴力猜测密码。

例2：在错误信息中出现SQL语句。这样攻击者一旦攻进数据库便会更加容易知道该查哪些表的哪些数据。

防范措施：检查报错内容，并且检查有没有关闭类似“Debug模式”的设置，不要暴露程序源代码或SQL这样的敏感信息。

例3：直接展示用户身份证号、手机号等敏感数据。

防范措施：如无必要展示则不展示，或者进行脱敏处理（138****1234、320101******121234）。
